# Техническое задание: бэкенд IT-Eco-For-SP

Документ для **бэкенд-разработчика**. Бэкенд проекта реализован как **Route Handlers** Next.js App Router (`src/app/api/**`) и серверная логика в `src/lib/**` с данными в **SurrealDB**. Аутентификация: **NextAuth** (JWT-сессии).

---

## 1. Цель

Обеспечить предсказуемое поведение API, согласованную модель прав (RBAC), целостность данных для **групп**, **событий (events)**, **профилей**, **админки** и внешних интеграций (Codeforces / AtCoder), с минимизацией дублирования проверок и регрессий.

---

## 2. Технический контекст

| Компонент | Расположение |
|-----------|----------------|
| API | `src/app/api/**/route.ts` |
| RBAC: роли и матрица | `src/lib/rbac.ts` |
| Обёртка для API | `src/lib/rbac/guard.ts` — `withRoleGuard`, `getSessionWithRole` |

Дополнительные требования по домену групп и событий: **`docs/ТЗ на группы.md`**.

---

## 3. Обязательные требования к реализации

### 3.1. Авторизация и RBAC на API

- Все эндпоинты, изменяющие критичные данные или отдающие персональные/агрегированные отчёты, должны явно проверять сессию и роль.
- **Предпочтительный паттерн:** оборачивать handler в `withRoleGuard` из `src/lib/rbac/guard.ts` с корректным `requiredRole: 'coach' | 'admin' | 'user'` (с учётом иерархии `hasRoleLevel`).
- Для сценариев «своя сущность / чужая» (группы, события) после прохождения `withRoleGuard` добавлять **доменную** проверку: coach видит только группы из `group_coaches`, user — только членство и т.д. (см. ТЗ на группы).
- Ответы при отказе: **401** — нет сессии; **403** — роль не подходит или запрет по домену; единый формат JSON с полем `error` (и при необходимости `ok: false`), согласованный с фронтом.

### 3.2. Согласованность с middleware

- Маршруты страниц, доступные без роли coach, но вызывающие «тренерские» API, не должны обходить проверки (сейчас **`/events`** не в `matcher` middleware — требуется решение: добавить matcher, серверные проверки на каждый метод, или редирект; зафиксировать в коде и в `docs/текущее-состояние-репозитория.md`).

### 3.3. Идентификаторы и запросы к SurrealDB

- Не смешивать в одном контракте API сырой id и `record<users>` без документации; использовать утилиты из `src/lib/surreal/ids.ts`.
- Запросы параметризовать; не конкатенировать пользовательский ввод в строку SurrealQL.

### 3.4. Миграции и схема БД

- Изменения схемы сопровождать скриптами в `docs/migrations/` (и при необходимости `.surql`), с кратким описанием порядка применения.
- Нативные `PERMISSIONS` в SurrealDB (см. `docs/migrations/001_rbac_schema.sql`): либо применить на стендах и убедиться, что клиент подключается с учётом политик, либо явно зафиксировать в README, что защита только на уровне приложения (и закрыть технический долг задачей).

## 5. Контракты API (требования к ответам)

- Успешные ответы по возможности в формате `{ ok: true, data: ... }` (или согласованный с фронтом исторический формат — не ломать без миграции клиента).
- Ошибки: `{ ok: false, error: string }`, HTTP-статус по смыслу.
- Для списков с пагинацией (если появится) — явные поля `cursor` / `limit` в контракте.

---

## 6. Интеграции

- **Codeforces / AtCoder:** учитывать лимиты внешних API; ошибки внешних сервисов не должны отдавать клиенту внутренние stack trace; при timeouts — понятный статус и retry-политика там, где уже принято в проекте (`queryWithRetry` и аналоги).
- **Email:** `nodemailer`; ошибки SMTP не должны приводить к 500 без логирования причины.

## 8. Критерии приёмки работ бэкендера

- Все новые и изменённые эндпоинты имеют явную проверку прав; нет «дыр» для coach/user на чужие `group_id` / `event_id`.
- Поведение согласовано с `docs/ТЗ на группы.md` и матрицей `docs/RBAC.md`.
- Миграции применимы на чистой копии схемы; есть инструкция порядка выполнения.
- Peer review: второй разработчик может пройти по чеклисту из `docs/TZ_TESTER.md` (раздел API) без расхождений с реализацией.
