# Подробное описание разделения ролей (RBAC) — IT-Eco-For-SP

> [!abstract] Стратегический обзор Архитектура управления доступом в проекте IT-Eco-For-SP построена на **принципе минимизации привилегий (POLP)**. Она обеспечивает изоляцию критической бизнес-логики (начисление рейтинга и кармы) от несанкционированного вмешательства, гарантируя прозрачность образовательного трека и защиту интеллектуальной собственности.

---

## Детализация ролевой иерархии (Модель 3+1)

Разделение прав жестко интегрировано в логику отображения интерфейса (Next.js 15) и валидируется на стороне сервера (SurrealDB).

### 1. Неавторизованный пользователь (Guest)

Роль предназначена для первичного ознакомления с платформой через публичный периметр.

- **Просмотр общего рейтинга:** Доступ к позициям участников в глобальном рейтинге для оценки соревновательной среды.
- **Просмотр грядущих соревнований:** Доступ к публичному календарю событий, который синхронизируется через изолированный Cron-воркер.
- **Маркетинг и авторизация:** Просмотр лендинга и возможность инициации процесса регистрации.

### 2. Участник (Participant)

Центральная роль, ориентированная на активное решение алгоритмических задач и соревновательную деятельность.

- **Решение задач:** Участие в активных контестах и использование фильтров для подгрузки задач с Codeforces.
- **Мониторинг прогресса:** Доступ к персональному Dashboard с графиками успеваемости, уровнем кармы и позицией в рейтинге.
- **Использование AI-ассистента:** Получение интеллектуальных подсказок по алгоритмам через систему RAG-адаптации контекста.

### 3. Тренер (Coach)

Экспертная роль, ответственная за управление образовательным контентом и мониторинг подопечных.

- **Управление контентом:** Создание и настройка параметров контестов.
- **Глубокая аналитика:** Доступ к агрегированным данным о производительности и активности вверенных групп пользователей.
- **Использование AI-ассистента:** Получение методических рекомендаций и отчетов о пробелах в знаниях группы на основе расширенного контекста данных.

### 4. Администратор (Admin)

Роль с максимальными полномочиями для технического и этического регулирования системы.

- **Управление пользователями:** Полный цикл контроля учетных записей (CRUD).
- **Корректировка показателей:** **Ручная корректировка кармы** на основе весовых коэффициентов за участие пользователей в офлайн-мероприятиях.
- **Технический надзор:** Модерация контента, контроль системных настроек и мониторинг работы Cron-воркера.
- **Использование AI-ассистента:** Полный доступ к функциям интеллектуального помощника для аналитики и управления.

##  Технические механизмы обеспечения RBAC

1. **Middleware (Next.js 15):** Обеспечивает превентивную проверку прав доступа на этапе рендеринга, гарантируя отображение только релевантных данных.
2. **SurrealDB Permissions:** Использование нативных разрешений на уровне таблиц (`DEFINE TABLE ... PERMISSIONS`) защищает данные от компрометации на стороне клиента.
3. **RAG-адаптация:** Механизм подмены промптов для AI-ассистента в зависимости от роли (помощь участнику vs аналитика для тренера).
4. **Изоляция воркеров:** Процессы обновления задач и календаря вынесены в **Render Background Worker**, чтобы гарантировать доступность данных о соревнованиях без нагрузки на основной API.
