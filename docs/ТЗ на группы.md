  ТЗ: «Группы участников» + мероприятия на группы + аналитика по группе



  1) Цель

  Добавить сущность групп участников, чтобы:
  • тренер/админ мог управлять группами и составом,
  • тренер мог назначать мероприятия (events) на группы, а не только на список отдельных
    пользователей,
  • тренер мог смотреть статистику по конкретной группе.


  2) Контекст текущего проекта (важное)

  • Стек: Next.js App Router + TypeScript + SurrealDB + NextAuth.
  • Роли: user, coach, admin (проверки в API через withRoleGuard).
  • Уже есть events:
    • visibility_type: public|private
    • participant_list: string[] (список userId для private)
    • API: GET/POST /api/events, PUT/DELETE /api/events/[id]
    • UI управления мероприятиями: src/app/(otherpage)/events/page.tsx
  • Есть синхронизация результатов private events: POST /api/events/[id]/sync-results (берёт
    participant_list, смотрит CF submissions, считает «карму»).
  • Важно по ID: в коде встречается смешение «сырой id» и users:.... В рамках этой фичи нужно
    стандартизировать и зафиксировать формат идентификаторов во всех новых полях/таблицах (см.
    раздел 4.4).


  3) Роли и права доступа



  3.1 Admin

  • CRUD всех групп.
  • Управление составом любой группы.
  • Назначение мероприятий на любые группы.
  • Просмотр аналитики по любой группе.


  3.2 Coach

  • CRUD только своих групп (где он назначен куратором/владельцем группы).
  • Управление составом только своих групп.
  • Назначение мероприятий только на свои группы.
  • Просмотр аналитики только по своим группам.


  3.3 User

  • Просмотр списка групп, в которых состоит.
  • Просмотр мероприятий, назначенных его группам.
  • Нет прав менять группы.


  4) Модель данных (SurrealDB)



  4.1 Таблица groups

  Поля:
  • id: record<groups>
  • name: string (обязательно)
  • description?: string
  • is_archived: bool default false
  • created_by: record<users> (владелец)
  • created_at: datetime, updated_at: datetime

  Требования:
  • Возможность архивировать группу (не удалять физически в MVP).


  4.2 Таблица group_members (состав группы)

  Поля:
  • group_id: record<groups>
  • user_id: record<users>
  • joined_at: datetime
  • (MVP) либо физическое удаление записи при исключении,
  • (опционально) история: left_at: datetime или status.

  Требования:
  • Один пользователь может состоять в нескольких группах.


  4.3 Таблица group_coaches (кураторы группы)

  Поля:
  • group_id: record<groups>
  • coach_id: record<users>
  • role_in_group: 'owner'|'coach' (опционально)
  • added_at: datetime

  Требования:
  • Поддержка нескольких тренеров на одну группу.
  • При создании группы текущий тренер автоматически добавляется как owner.


  4.4 Формат ID (обязательное решение)

  Нужно выбрать единый формат:
  • вариант A: везде хранить users:xxxx, groups:yyyy (Surreal thing id в строке),
  • вариант B: хранить «сырой id» без префикса и всегда преобразовывать в запросах.

  В ТЗ для реализации рекомендовать вариант A: хранить users:... / groups:... в новых полях и
  join-таблицах, чтобы не ломать type::thing() и сравнения.


  5) Расширение events: назначение на группы



  5.1 Новые поля в events

  Добавить:
  • target_groups?: string[] — список groups:... (только для private)
  • (опционально) participant_snapshot?: string[] — зафиксированный список users:... для
    результатов

  Сохранить обратную совместимость:
  • participant_list остаётся и работает как «точечные пользователи».


  5.2 Правила видимости private event

  Private event виден пользователю если:
  • он перечислен в participant_list, или
  • он состоит хотя бы в одной группе из target_groups.


  5.3 Снапшот участников для результатов (чтобы состав группы не менял итоги задним числом)

  Правило:
  • перед начислением результатов (например, при sync-results или при переводе status=completed)
    вычислить эффективный список участников:
    • participant_list ∪ members(target_groups)
  • сохранить его в participant_snapshot
  • дальнейшие расчёты/повторные синки использовать participant_snapshot


  6) API (минимально необходимое)



  6.1 Groups

  • GET /api/groups
    • coach: только свои (где есть запись в group_coaches)
    • admin: все
    • user: только где состоит (через group_members)
  • POST /api/groups (coach/admin)
  • GET /api/groups/[id] (доступ по правилам ролей/принадлежности)
  • PATCH /api/groups/[id] (coach только свои; admin любые)
  • DELETE /api/groups/[id] (в MVP лучше: is_archived=true)


  6.2 Members

  • GET /api/groups/[id]/members
  • POST /api/groups/[id]/members body: { user_ids: string[] }
  • DELETE /api/groups/[id]/members body: { user_id: string } (или параметр)


  6.3 Analytics

  • GET /api/coach/groups/[id]/analytics?from&to
    • доступ: coach только свои группы; admin любые


  7) Изменения существующих API events

  • POST /api/events и PUT /api/events/[id]:
    • если visibility_type=private разрешить target_groups
    • валидировать: private должен иметь хотя бы participant_list или target_groups (или оба)
  • GET /api/events и GET /api/events/my:
    • расширить выдачу private events по членству в группах


  8) UI/UX



  8.1 Раздел тренера: «Группы»

  Страницы (предложение):
  • /coach/groups — список групп, создание, архивирование
  • /coach/groups/[id] — состав группы (поиск пользователей, добавление/удаление)
  • /coach/groups/[id]/analytics — аналитика по группе


  8.2 Форма создания/редактирования private event

  В src/app/(otherpage)/events/page.tsx добавить:
  • multi-select групп (с поиском)
  • оставить выбор отдельных пользователей
  • показывать «эффективное количество участников» (union из выбранных групп + отдельных
    пользователей)


  9) Аналитика по группе (MVP-метрики)

  Минимально:
  • кол-во участников
  • средний bscp_rating (опционально медиана)
  • распределение платформ: CF / AtCoder / обе / нет (по external_accounts)
  • участие в мероприятиях за период (сколько событий назначено группе, сколько завершено)
  • карма/результаты по завершённым group-events (если есть данные из sync-results)

  Примечание: если сейчас нет таблицы для хранения результатов синка, в MVP можно:
  • показывать «готово/не готово синхронизировано» и список участников, кому начислялось,
  • или добавить отдельную таблицу event_results/event_sync_logs (опционально, если нужно
    сохранять историю).


  10) Критерии приёмки

  • Тренер создаёт группу и управляет составом.
  • При создании private мероприятия тренер выбирает 1+ групп; участники группы видят мероприятие
    в списках.
  • sync-results корректно работает для мероприятий, назначенных через группы (через снапшот).
  • Тренер видит страницу аналитики по группе с MVP-метриками.


  11) Риски/особые случаи

  • Пользователь в нескольких группах: учитывать union без дублей.
  • Изменение состава группы после завершения события: результаты не должны «переехать» → нужен
    participant_snapshot.
  • Стандартизация ID: обязателен единый подход, иначе фильтрация видимости/участники могут не
    совпадать.
